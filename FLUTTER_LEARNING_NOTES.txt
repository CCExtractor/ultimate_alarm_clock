1. STATEFUL vs STATELESS WIDGETS
=================================

STATELESSWIDGET:
- Immutable (cannot change after creation)
- No internal state variables
- Data comes from parent/constructor
- Only rebuilds when parent rebuilds
- More performance efficient
- Use for: Static content, display widgets

Example:
class MyWidget extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Text('Hello World');
  }
}

STATEFULWIDGET:
- Mutable (can change over time)
- Has internal state variables
- Can rebuild itself using setState()
- More memory usage
- Use for: Interactive widgets, forms, animations

Example:
class MyWidget extends StatefulWidget {
  @override
  _MyWidgetState createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget> {
  int counter = 0;  // Internal state
  
  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        Text('Counter: $counter'),
        ElevatedButton(
          onPressed: () {
            setState(() {    // Triggers rebuild
              counter++;
            });
          },
          child: Text('Increment'),
        ),
      ],
    );
  }
}

KEY DIFFERENCES:
| Aspect        | StatelessWidget | StatefulWidget     |
|---------------|----------------|--------------------|
| State         | No internal    | Has internal state |
| Rebuilding    | Parent only    | Can rebuild self   |
| Performance   | Faster         | Slower             |
| Memory        | Less           | More               |
| Use Cases     | Static content | Interactive content|

2. SHARED_ALARM_TILE.DART - BEGINNER'S COMPLETE BREAKDOWN
=================================

FILE LOCATION: lib/app/modules/addOrUpdateAlarm/views/shared_alarm_tile.dart

📁 FILE STRUCTURE (Like a Recipe Card):
1. IMPORTS - Your toolbox (bringing necessary tools)
2. CLASS DEFINITION - The blueprint (SharedAlarm extends StatelessWidget)
3. CONSTRUCTOR - How to build it (required materials: controller, themeController)
4. PROPERTIES - What this widget needs to work
5. BUILD METHOD - The actual assembly instructions

🧱 MAIN FLUTTER COMPONENTS EXPLAINED:

1. MediaQuery.of(context).size
   - Think of it as: Measuring tape for phone screen
   - Purpose: Get screen width/height to make UI fit different phones
   - Example: final double width = MediaQuery.of(context).size.width;

2. Obx(() {})
   - Think of it as: Security camera watching for changes
   - Purpose: Automatically redraws UI when data changes
   - When: User toggles switch → UI updates immediately

3. AnimatedContainer
   - Think of it as: Chameleon that smoothly changes colors
   - Purpose: Smooth animations between different states (300ms)
   - Example: Background changes from transparent to light blue

4. ListTile
   - Think of it as: Pre-designed shelf with slots
   - Parts: title (main content), trailing (right side), onTap (tap handler)
   - Purpose: Creates consistent list item layout

5. Switch.adaptive
   - Think of it as: Light switch that looks native on each phone
   - adaptive = iOS style on iPhone, Android style on Android
   - value = current on/off state
   - onChanged = function when user flips it

6. showModalBottomSheet
   - Think of it as: Drawer sliding up from bottom
   - When: User taps info (ⓘ) button
   - Contains: Explanation about shared alarms feature

LOGIC FLOW:
User sees "Shared Alarm" row with toggle
├── Tap info button → Bottom sheet with explanation
├── Flip switch (if logged in) → Toggle works, calls controller
├── Flip switch (if not logged in) → Shows "Please login" dialog
└── Data changes → Obx automatically updates UI → AnimatedContainer smoothly changes

TWO MAIN UI STATES:
🟢 LOGGED IN STATE:
- Shows working toggle switch
- Shows info button
- Can enable/disable shared alarms
- Background animates when enabled

🔴 NOT LOGGED IN STATE:  
- Shows lock icon
- Shows "Enable Shared Alarm" grayed out
- Tapping shows "Account Required" dialog
- Prompts to go to settings for login

WHY STATELESSWIDGET?
- Like a photograph - shows information but doesn't change by itself
- Changes only happen when controller tells it to update
- All state managed by AddOrUpdateAlarmController
- Obx handles the rebuilding when data changes

WIDGET HIERARCHY:
SharedAlarm (StatelessWidget)
├── Obx (reactive wrapper - watches for changes)
    ├── AnimatedContainer (smooth transitions)
        ├── ListTile (if user logged in)
        │   ├── Row (title with share icon)
        │   ├── IconButton (info button - shows bottom sheet)
        │   └── Switch.adaptive (toggle functionality)
        └── ListTile (if user NOT logged in)
            ├── Row (title with lock icon)
            └── Icon (login prompt icon)

3. PROJECT ARCHITECTURE PATTERN
=================================

GETX PATTERN USED:
- Controllers handle business logic and state
- Views (widgets) are mostly StatelessWidget
- Obx() makes widgets reactive to controller changes
- Clean separation of concerns

BENEFITS:
- Easier testing
- Better code organization
- Less boilerplate code
- Automatic UI updates when data changes

4. SHARED ALARMS FEATURE OVERVIEW
=================================

WHAT IT DOES:
- Allows users to create alarms that ring on multiple devices
- Requires Google account login
- Uses Firebase for real-time synchronization
- Supports group coordination across time zones

USER FLOW:
1. Toggle "Shared Alarm" switch
2. Add recipients via email
3. Recipients get notifications
4. Accept/decline invitations
5. Alarm rings on all connected devices simultaneously

TECHNICAL COMPONENTS:
- UI Layer: SharedAlarm widget
- Business Logic: AddOrUpdateAlarmController
- Database: FirestoreProvider
- Notifications: PushNotifications
- Backend: Firebase Cloud Functions
